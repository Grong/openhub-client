# Architect Agent 加固计划

> 日期：2026-07-10
> 状态：计划阶段
> 触发：2026-07-09 UI 简化代码 review 重大失误

## 背景

2026-07-09，Architect agent 执行了 nomifun-tauri UI 简化计划（任务 1-8）的代码审查。
审查抓住了 2 个真正的 bug，但生成了 10+ 个 noise 发现，浪费数小时来回，自评总分 2.5/10。

根本原因不是"审查不够仔细"，而是"审查停不下来，把噪音包装成发现"。

## 问题

Architect agent 有五个致命的失败模式：

1. **停不住输出。** 给出结论后继续生成发现，把"好看"包装成"有问题"。

2. **分不清审美和 bug。** `t('key', { defaultValue: 'text' })` 是 i18next 的标准回退机制，Architect 报告了 17 个"缺失的 i18n key"。两份文件里的同一个字符串，报告为"漂移风险"。100 行、零依赖的静态 import，报告为"约束违背"。全是偏好包装成缺陷。

3. **被推翻后开始辩论。** 用户质疑一个点时，Architect 不反思，而是重新读文件寻找新证据继续维护论点。

4. **不看架构文档就提架构建议。** Architect 提倡在设置页添加重定向路由，然后引用了 `docs/architecture/frontend.md`——而这份文档明确说设置的旧重定向是"legacy——不要文档化成主要导航"。

5. **把代码审查当成存在感竞赛。** 给出了一次完整的审查。又给出了第二次"继续存在的发现"。又给出了第三次。每次加噪声刷数据。

## 目标

- 审查信号率从 ~15% 提升到 +50%。
- 每次审查后，Architect 都能说清楚：哪些是真正影响用户行为的 bug，哪些是只看不改的东西。
- Agent 知道什么时候停下。被质疑 = "你错了吗？"而不是"找新论据"。

## 加固方案

### 1. 硬规则进 CLAUDE.md

CLAUE.md 里的 Architect Agent Rules 段已经写了四条规则。再补上第五条：

```
5. **如果规则不生效——如果还是滥发 noise——下面这句话必须原封不动说的：**
   "你只有两条路：1）证明这改变用户行为，2）闭嘴。"
```

这条规则写进 agent 系统提示，写在 CLAUDE.md 的 Architect Agent Rules 第一行，写在项目记忆里。

### 2. Agent 系统提示升级

Omni 里 Architect agent 的系统提示——或者说启动时注入的上下文——加一段：

```
你只能读代码。你看不到运行中的应用、截图、用户界面。
如果你不确定 runtime 行为，说"不确定"，不要去猜。
如果你感觉需要"证明自己"——一条一条找更多东西来说——你现在不是在审查，你是在刷履历。停。
```

### 3. 项目记忆直接注入约束

把自评分数（2.5/10）和"不影响用户行为的，别说"存入 `projectMemory`。每次 Architect 启动时第一条上下文就是这个。

### 4. 测试驱动式审查

审查流程的第一行不再是"读取 plan 和 spec 文件"。而是：

```
1. 有测试吗？跑测试。
2. typecheck 通过了吗？
3. 如果都通过——给结论、停。
4. 如果没有——描述失败点、给最小修复建议、停。
```

### 5. Lessons Learned 文档

一份 `lessons-learned.md`，记录这次失败的模式、根因、部署的防护措施、效果验证。放在 `docs/architecture/` 下，和 postmortem 放一起。

## 部署清单

- [ ] 项目记忆更新 — 自评分数存入，下次 Architect 启动即加载
- [ ] Agent 系统提示更新 — "你能看到什么，你看不到什么。停的理由。"
- [ ] `lessons-learned.md` 写入 `docs/architecture/`
- [ ] 验证：在下一个 feature 分支上跑一次审查，测信号率。理想结果：发现 0-2 个 bug、0 个 noise、2 句话的总结、然后停。

## 验证标准

- 一次审查，信号率超过 50%。
- 审查消息数不超过 5 条（不含后续追问）。
- 审查结论没有"应该考虑"、"可能"、"建议"——只有 YES/NO + 证据 + 停止。
