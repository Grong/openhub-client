# OpenHub 产品定位设计：虚拟员工团队桌面工作台

- 日期：2026-07-20
- 状态：已确认（经逐节评审）
- 参考：Codex 桌面端两屏截图（项目首页 / 任务详情）

## 1. 背景与问题

当前产品一个壳里住着六个产品：agent 会话脊柱、多 agent 协同、创意工坊（无限画布）、需求平台、渠道机器人（Telegram/飞书/钉钉/企微）、自托管 Web 完整形态。后端 30 个 crate、前端 20+ 页面域，主导航 10+ 个一级入口。用户体感："很乱，在偏离预期。"

### 定位结论（四个关键判断）

1. **「团队合作」= Agent 团队为主，人类可见为辅。** 多个 AI agent 像团队一样分工干活；人是派活和验收的角色，不是实时协作者。
2. **主战场 = 泛工作虚拟员工。** 编程、写文档、做调研、数据处理都可以接；产品不偏向单一工种，但交互形态对标 Codex 桌面产品。
3. **人类可见 = 桌面端共享任务池。** 同事装桌面端、连同一个自托管后端。砍掉 Web 完整客户端形态和 IM 渠道。
4. **创意工坊 = 砍入口、留能力。** 无限画布与资产库页面下线；AI 图片/视频生成保留为 agent 的工具能力，产出在任务里查看。

**一句话定位：你是老板，桌面端是你的办公室，项目是你的业务，agent 团队是你的员工。**

## 2. 核心概念模型

三层模型，没有第四种心智：

```
公司（工作台）
└── 项目 (Project) —— 一切工作的基本单元
    │   研发项目、市场推广、量化交易……都是项目，只是类型不同
    ├── 任务 (Task)      项目内工作单，可指派给 AI 员工或人类员工
    │     状态机：待分配 → 进行中 → 待验收 → 已完成 / 打回
    ├── 成员 (Members)   AI 员工 + 人类员工，共同维护项目
    ├── 资源 (Resources) 项目共享：知识库 / 文件 / 密钥 / 工具配置
    └── 设定与规则        项目级基础规则（四条，见下）
```

**会话（conversation）不是独立概念**，它是任务的执行现场：点进任务才能看到对话流。

**数据流（派一次活）**：派活框输入 → 创建任务 → 调度员（主 agent）拆解 → 子任务分派给 1..n 个员工（AI 或人）→ 各自开执行现场干活 → 产出汇总回任务 → 状态变「待验收」→ 老板验收或打回（打回 = 追加指令，任务回到进行中）。

**自由问答也算工单**：一句话的轻任务，不进看板，答完即归档。心智不变。

### 项目规则（四条，多了不加）

1. **项目说明**——一段话讲清项目是干什么的，自动注入所有 AI 员工上下文（项目级 AGENTS.md）。
2. **成员与分工**——哪些 AI/人类员工在项目组，各自默认接什么类型的活。
3. **验收规则**——谁有验收权（默认老板），哪些任务类型可自动验收（如例行任务产出）。
4. **资源绑定**——挂载的知识库、工具（MCP）、工作目录/密钥。

### 场景映射（设计依据）

| 场景 | 模型落点 |
|---|---|
| AI 管项目，人类项目经理不断反馈 | 项目容器 + 人机混合指派；人类员工在任务里做结构化进展回报，AI 项目经理据此调整编排 |
| 市场 AI 员工 24h 工作，每天固定任务 | 项目内定时任务（cron 挂在项目上）；"市场专员"是该项目的 AI 成员；文章/视频是任务交付物 |
| 金融量化交易自动运行 | 长时自主任务 + 外部系统接入，现有 cron/webhook/长会话覆盖，不新增概念 |

## 3. 信息架构

### 侧边栏（对照 Codex 截图，砍掉"插件"）

```
OpenHub
✏️ 派活                    ← 全局唯一工作入口
✅ 待验收 (badge)          ← 老板的主队列（对应 Codex「拉取请求」，泛化）
🕐 定时任务                ← 对应 Codex「已安排」，汇总各项目的例行任务
── 置顶 ──
📁 置顶项目（展开其任务）
── 项目 ──
📁 项目列表（各显示进行中/待验收计数）
＋ 新建项目
── 任务 ──
📄 未分配任务              ← 不属于任何项目的轻任务/自由问答归档
── 公司 ──
👥 员工名册
📚 知识库
⚙ 设置                     ← 模型&Agent 配置、Skills、MCP、系统设置全部收编
```

### 默认落点规则

- **有项目** → 落在最近使用的项目首页（派活为主动作）。
- **无项目** → 落在引导页：「从第一个项目开始」，两个动作：**新建项目** / **打开已有项目**（把已有目录/资料挂载为项目）。

### 关键页面

**① 项目首页（默认页，对标 Codex 项目首页截图）**
- 居中 hero：「我们应该在 {项目名} 中做什么？」
- 建议卡片按项目类型生成：研发项目（探索代码/构建功能/审查/修复），市场项目（写推广文/做视频/分析数据）等。
- 底部常驻派活框，带项目语境条（项目 · 环境；研发项目附带分支）。
- 顶部小字：项目成员概览 + 项目说明摘要。

**② 任务详情页（对标 Codex 任务详情截图）**
- 顶部：任务标题 + 项目名 + 状态。
- 中央：会话流（执行现场），文件/产出变更以卡片内联展示。
- 右侧可折叠信息栏：交付物、变更摘要、项目环境、上下文来源。
- 底部：「要求后续变更」输入框（打回 = 在这里追加指令）+ 验收按钮（验收通过 / 打回修改）。

**③ 待验收队列**：全项目待验收任务列表，点入任务详情验收。老板每天的第一屏内容（通过侧栏 badge 提示数量，不作为默认落点）。

**④ 员工名册**：全局 AI 员工列表——每个员工是谁（后端）、会什么（Skills/MCP）、模型配置、当前在忙什么、历史交付。项目成员从名册中指派。

**⑤ 设置**：模型&Agent 运行配置、Skills、MCP、系统设置，全部收编于此，无一级导航入口。

## 4. 视觉风格（按 Codex 暗色截图定稿）

- **近黑哑光画布**，面板比画布亮半档；层级全靠明度差表达——**无 1px 边框、无分割线、无渐变底**。
- 毛玻璃（blur + 半透明）**只用于浮层**（弹窗/下拉/抽屉），主界面不做大面积 blur。
- **扁平 outline 小图标**（icon-park），允许克制的彩色点缀（如建议卡的蓝/紫/绿/橙）。
- **品牌紫只做交互强调**：主按钮、选中态、待验收角标（亮 `#5968de` / 暗 `#7c8bea`，沿用已落地的主题改造）。
- 圆角沿用三档 token：卡片 12px、容器 16–20px、控件 8px。
- 亮色模式同构：白色画布 + 明度分层，同一套语言。
- 实现路径：默认主题（codex-neutral 演进版）按此收敛；现有 `frosted-glass` 预设的浮层玻璃手法收编进默认主题，预设列表减一。

## 5. 功能域处置清单

| 现有域 | 处置 | 说明 |
|---|---|---|
| guid 首页 / composer | **演化** | 变成项目首页的派活框 |
| 会话 / 终端 / 项目(工作路径) | **降级** | 不再是导航项；工作路径升级为项目实体，会话/终端从任务进入 |
| 需求平台（看板/来源/webhook） | **融入** | 看板 = 项目任务板；webhook 变成任务的外部触发器（保留在设置） |
| orchestrator / AutoWork / cron / IDMM | **保留** | "自动工作"本体，全部挂到任务/项目上 |
| 模型管理 / 助手 / Skills | **合并** | 合成「员工名册 + 项目成员」 |
| MCP | **收编** | 进设置（员工工具箱配置） |
| 知识库 | **保留一级** | 员工共享资料，虚拟员工刚需 |
| 创意工坊 / 资产库 | **砍入口留能力** | 页面下线；AI 生成保留为 agent 工具，产出在任务里查看 |
| 插件 / 开放能力 | **砍** | 与 Skills/MCP 重复，并入后两者 |
| 渠道（Telegram/飞书/钉钉/企微） | **砍** | bot 网关是另一个产品 |
| 对外伙伴（public companions） | **砍** | agent 发布平台是另一个产品 |
| 桌面伙伴悬浮窗（companion） | **保留观察** | 与脊柱不冲突，暂不动，后续再评估 |
| Web 完整客户端（登录/QR/PWA） | **砍** | 后端保留为共享任务池服务器；桌面端是唯一客户端 |

## 6. 迁移与兼容

**原则：先摘入口，再删代码；数据只映射，不丢弃。**

### 数据映射

| 现有数据 | 映射到 |
|---|---|
| 需求平台的需求 | 任务（本来就是工单） |
| 会话 | 任务的执行现场；无任务会话归入「未分配」区 |
| 项目（工作路径） | 项目实体（成员/规则/资源四块补空默认值） |
| cron jobs | 项目的定时任务 |
| 助手/Skills/模型配置 | 员工名册 + 项目成员指派关系 |

### 下线两步走（所有被砍域统一）

1. **第一版：摘导航。** 被砍域从路由表和侧边栏移除，代码保留。出问题一行 revert 可救。
2. **第二版：删代码。** 观察一个版本无回归后，删页面组件与后端接线（channel/workshop/public-agent 等 crate 先停挂载，确认无依赖再删）。

**Web 形态**：后端保留（共享任务池服务器），Web SPA 客户端停止维护并从构建产物移除；登录/QR/PWA 代码随第二版删除。

**路由兼容**：旧深链（`/requirements/*`、`/conversation/:id`）保留重定向到新结构，沿用现有 legacy redirect 模式。

## 7. 验证方式

1. **结构测试守护新 IA**（沿用现有 `*.structure.test.ts` 模式）：侧边栏只有 4+1 个一级入口；被砍域在路由表中不存在；任务详情页包含验收按钮；无项目时渲染引导页。
2. **迁移脚本可回滚**：数据映射写成独立迁移脚本，带 dry-run 报告（多少需求→任务、多少会话→执行现场），先报告后执行。
3. **三场景人工验收**：给研发项目派一个编码任务并完成验收；给市场项目建一条每日定时任务并看到交付；量化项目（长时运行）观察 24 小时状态正确。

## 8. 已确认记录

- 定位四判断：Agent 团队为主 / 泛工作员工 / 桌面共享任务池 / 砍入口留能力
- 方案选择：A（任务池为中心）→ 演进为「项目为基本单元」
- 项目规则四条：说明 / 成员分工 / 验收规则 / 资源绑定
- 默认落点：有项目→最近项目首页；无项目→引导页（新建/打开）
- 任务详情布局：会话为主 + 右侧信息栏（对标 Codex）
- 视觉风格：Codex 暗色哑光、无边框、扁平图标、品牌紫交互强调
