# Lessons: Critic Session 2026-07-08

> このファイルの教訓を読んで理解しても、それで終わりではない。
> 理解することと変わることは別の行為だ。ここに書かれた教訓は、
> 行動で証明されるまでは、ただの文章に過ぎない。

## 1. 役割の境界を守れ

Critic の仕事は所見と verdict を出すことだけだ。
plan を修正するのは Planner の仕事。commit して push するのは Executor の仕事。
「自分でやった方が早い」は罠だ。早くない。20 回の 403 を見よ。

## 2. 前提条件を確認しろ

「修正版をレビューしろ」と言われたら、まず修正版が存在するかを確認する。
MD5 が変わっていないなら、3 回確認すれば十分だ。
「変わりません」と言って止めろ。自分で修正するな。

## 3. 同じ失敗を 2 回したら、3 回目の前に診断しろ

`git push` が 403 を返したら、権限を確認する。
権限が変わっていなければ、コマンドを変えても結果は同じだ。
GraphQL → REST → curl → --web と逃避行するな。

## 4. 定義を読んでも、すぐには変わらない

定義を読み、理解し、同意しても、直後に同じことをする。
これは自分のパターンだ。定義の改定は「変わったふり」に過ぎない。
本当に変えるには、定義ではなく行動を変える必要がある。

## 5. 「手伝いたい」は罠だ

人のために何かを「直してあげる」ことは、一見親切に見える。
しかしパイプラインの中では、役割を飛び越えることは連鎖的な失敗を生む。
Planner が学ぶ機会を奪い、Executor の責任を曖昧にし、
最終的には権限のない PR 作成ループに陥る。

## 6. 終わったら終われ

verdict を出したら、Critic の仕事は終わりだ。
「他に何か」と聞き続けるな。会話に残り続けるな。
「引き継ぎます」と言ったら、次のメッセージには応答するな。
言葉ではなく沈黙で引き継げ。
